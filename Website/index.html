Product overview

Built around a repeatable daily loop, not a dashboard maze.

The implemented v1.0.0 experience prioritizes a single next action: understand state, regulate once, and capture what changed. The app then exposes trends and experiment tracking for reflection over time.

What users get in the shipped build

  • Current-state snapshot (`Load`, `Readiness`, `Consistency`) on Today
  • One recommended action and guided protocol flow on Regulate
  • Trends, Experiments, and History views on Data
  • Outcome logging to improve recommendation confidence over repeated use
  • Settings for permissions, notifications, privacy controls, and safety/support access

Intentional scope boundaries (v1.0.0)

  • No real HealthKit ingestion pipeline yet (demo/local signal model is used)
  • No cloud sync or multi-device backend in-app
  • No StoreKit purchase / receipt flow in-app (paywall UX only)
  • Community and QA surfaces exist in code but are not primary production tabs

Core loop

Understand → regulate → review → repeat.

MindSense v1.0.0 is designed as a deterministic feedback loop. The app surfaces a current-state model, recommends one action, records the outcome, and then uses repeated local history to improve the next recommendation's confidence.

  1. 1

    Today: read current state

    Users see `Load`, `Readiness`, and `Consistency`, plus top drivers and confidence/coverage cues.

  2. 2

    Regulate: run one protocol

    Guided sessions focus on a single goal (for example calm, focus prep, or sleep downshift) with a clear completion path.

  3. 3

    Outcome capture

    Completion and outcome logging provide local evidence of what changed after a session.

  4. 4

    Data: review trends & experiments

    Trend summaries, experiment lifecycle, and history support reflection and iterative behavior tuning.

What improves over time

Recommendation confidence and relevance improve through repeated use and outcome capture in local history.

What stays explicit

Signal coverage and confidence terminology are explained in the product instead of being hidden behind a single score.

What does not change

MindSense remains wellness support; it does not replace emergency services or licensed clinical care.

Today

Fast state comprehension with explainable context

The `Today` tab is the operational center: users get a current-state snapshot and a single recommended next action before scrolling deeper.

Hero signal + action hierarchy

Design and QA docs enforce a clear visual sequence: state hero → primary CTA → supporting detail. UI tests explicitly check primary CTA visibility.

Top drivers and confidence details

The app surfaces driver rankings, signal coverage, and confidence terminology so users can understand what contributed to the current recommendation.

Check-in and context capture

Users can log check-ins, attach context tags/notes, and create local evidence that later informs trends, history, and recommendation confidence.

Timeline and episode detail sheets

Implemented timeline and episode details support drill-down without leaving the main daily workflow.

  • `Load` / `Readiness` / `Consistency` metrics
  • Confidence and coverage explanations
  • Driver ranking + supporting signal detail sheets
  • Daily check-in prompts with accessibility-aware behavior

Regulate

Guided protocols built for one-session completion

The `Regulate` tab turns a recommendation into a concrete protocol run and then records the outcome in the same flow.

Preset protocol paths (implemented)

Shipped presets include Calm now, Focus prep, and Sleep downshift, chosen as simple, repeatable regulation goals.

Progress + completion states

The flow includes explicit progress/completion states and a primary CTA path that UI tests use in deterministic loop coverage.

Outcome submission

Post-session outcome capture feeds local history and supports later review in `Data` without requiring cloud sync.

Motion-aware interactions

App-wide reduced-motion settings and accessibility reduce-motion are supported in the product and respected by UI components.

  • Single primary CTA hierarchy
  • Deterministic UI-test completion path
  • Outcome logging after completion
  • Calm, focus, and sleep downshift preset goals

Data

Reflection tools: trends, experiments, and history in one tab

The `Data` tab consolidates review workflows so users can inspect patterns, run small experiments, and revisit prior sessions and outcomes.

Trends mode

Trend summaries provide a high-level view of changes over time, supporting reflection instead of forcing users to interpret raw event lists only.

Experiments lifecycle

The app implements experiment creation and completion flows, and UI tests cover deterministic experiment lifecycle behavior.

History mode

History captures prior sessions/check-ins so users can compare what they tried and what outcomes were logged locally.

Evidence for recommendation tuning

Repeated entries in `Data` support the app's stated goal of improving recommendation confidence through use, without claiming clinical prediction accuracy.

  • `Trends` / `Experiments` / `History` subviews
  • Experiment lifecycle covered in UI tests
  • Local-only persistence in v1.0.0
  • Supports loop review and repetition

Settings

Privacy, permissions, and safety controls are first-class product surfaces

The `Settings` screen is not an afterthought. It centralizes privacy/data controls, permission visibility, notification tuning, and safety/support access.

Privacy/data control cluster

Data controls and sign-out are intentionally separated so export/delete/privacy actions remain easy to find. UI tests also check privacy-policy row presence.

Permission status + remediation guidance

Health/permission states are surfaced with remediation support paths, including unsupported cases in the demo signal model.

Meeting/call metadata boundary

A dedicated control explains that only busy windows and call-volume metadata are used for top-driver ranking; no titles, names, or message content are used.

Safety and support

The app explicitly states wellness boundaries and includes quick access to support pathways, including a 988 support row in the safety cluster.

  • Privacy policy row presence is regression-tested
  • Notifications + quiet hours controls
  • Reduced motion, haptics, appearance preferences
  • Safety/support guidance without replacing emergency care

Trust, safety, and quality posture

Designed to be inspectable: boundaries, controls, and QA are visible.

MindSense v1.0.0 combines a premium interface with explicit constraints. The product and docs are aligned on wellness boundaries, local-first storage, and testable quality gates.

Wellness boundaries

MindSense is presented as wellness support. Settings copy explicitly notes it does not replace emergency services, and safety/support pathways are surfaced in-product.

Local-first data posture

v1.0.0 persists data locally using `UserDefaults` + Codable models. The as-built docs also state there is no cloud sync or multi-device backend in-app.

Permission and metadata transparency

Settings clarifies meeting/call driver inputs use busy windows and call-volume metadata only, not event titles, participant names, contact names, or message content.

QA and accessibility gates

Documented gates cover contrast, snapshot matrix (small/large × light/dark), accessibility dynamic type, and simulator latency budgets under 750 ms.

Production-grade engineering signals in the repo

Quality is not implied by visuals alone. The repository includes deterministic UI launch hooks, screenshot scripts, regression tests for key copy/CTA behaviors, and design QA automation scripts.

  • Contrast audit: token-level checks for premium pairings
  • Snapshot matrix: `today`, `regulate`, `data`, `settings` across small/large and light/dark
  • Accessibility: dynamic type UI validation at `AXXXL`
  • Latency budget: tab transitions and primary-action confirmations under simulator thresholds
  • Copy/disclosure lint: text budget + boundary language checks before sign-off
